# 03. 데이터 모델

엔티티별 필드 정의. 실제 스키마는 `server/src/entities/`의 TypeORM 엔티티가 기준이며, 이 문서는 그 요약이다.

## 사용자·팀

### User
`id`, `kakao_id`, `kakao_email`, `name`, `university`, `department`,
`profile_image_url`(카카오 프로필), `email_opt_out`(알림 메일 수신 거부),
`is_deleted`(탈퇴 — 행 내 익명화)

> 탈퇴는 soft delete(`deleted_at`) 대신 **행 내 익명화**(`is_deleted=true`, 이름·카카오 식별자·이메일·프로필·학교 정보 제거) — soft delete를 쓰면 TypeORM 필터가 과거 리포트의 사용자 조인을 null로 만들기 때문. ([04](04-API-명세.md) 회원 탈퇴)

### Team
`id`, `name`, `course_name`, `invite_code`, `created_by`, `deleted_at`(soft delete — 마지막 멤버 탈퇴·팀 삭제 시, 회의·기여도 데이터 보존)

### TeamMembership
`team_id`, `user_id`, `role`(leader/member), `joined_at`, `deleted_at`(soft delete — 탈퇴/추방 이력 보존)

### TeamSettings
`team_id`, `punctuality_grace_ratio`, `presence_grace_seconds`, `max_utterance_chars`, `deadline_penalty_curve`(standard=0.2/일 · lenient=0.1/일 · strict=초과 즉시 0), `absent_meeting_handling`(exclude/zero/attendance_only), `min_meeting_minutes`(기본 5, 누적 반영 최소 회의 길이), `final_task_weight`(기본 0.50, 최종 합산의 테스크 비중 — 그룹 생성 시에만 설정·이후 불변), `leader_bonus_multiplier`(기본 1.00, 팀장 최종 점수 배율, 1.0 캡), `contribution_visibility`(team=전체 공개 / self=본인만 / leader=팀장만 전체 열람)

> 세부 가중치(발언:참석, 액션 난이도)는 **전역 고정**(팀별 조정 불가) — `contribution_weights` 없음. 단 최종 합산의 테스크:회의 비율(`final_task_weight`)만 그룹 생성 시 설정 가능. ([06](06-기여도-산정.md))
>
> **변경 이유** — "세부 가중치는 고정, 팀별 차이는 설정값으로만" 원칙 적용. `contribution_weights` 제거(3축 가중치 전역 고정). `final_task_weight`(테스크:회의 비율, 생성 시 고정)·`leader_bonus_multiplier`(팀장 부담 보정)·`min_meeting_minutes`(짧은 회의가 누적 평균 왜곡 방지)를 추가.

## 프로젝트·목표

### Project
`id`, `team_id`, `title`, `status`(active/archived)

### Milestone
`id`, `project_id`, `title`, `deadline`, `order_index`, `progress_ratio`

> 마일스톤 진척도는 조회 시 동적 계산해 응답에 포함 — API는 팀이 아닌 **프로젝트 하위**로 묶인다. ([04](04-API-명세.md) 프로젝트·마일스톤)

## 회의

### Meeting
`id`, `team_id`, `scheduled_at`, `total_minutes`, `topic`,
`status`(scheduled/active/ended), `t0_timestamp`,
`ended_at`(실제 종료 시각 — 출석 분모 = `ended_at − t0_timestamp`),
`meeting_type`(regular/partial/test — regular만 산정, test는 산정하되 누적 제외),
`is_invalidated`(무효 처리 — 누적(②) 집계서 제외),
`summary`(회의 후 AI 종합 정리로 생성)

> **변경 이유** — `ended_at`: 출석률 분모를 예정 시간이 아닌 **실제 진행 시간**으로 잡아 조기 종료·연장에도 공정하게. `meeting_type`: 정규 회의만 점수 반영, 부분은 미산정·테스트는 누적 제외하려 회의 종류 구분.

### Agenda (★)
`id`, `meeting_id`, `title`, `estimated_minutes`, `order_index`,
`status`(pending/active/done), `started_at_offset_ms`, `ended_at_offset_ms`,
`actual_minutes`, `source`(ai_recommended/manual/ad_hoc), `summary`(완료 시 LLM 생성), `milestone_id`

### Utterance
`id`, `meeting_id`, `user_id`, `text`, `char_count`, `confidence`,
`started_at_offset_ms`, `ended_at_offset_ms`, `agenda_id`(★ 안건 매칭)

## 회의 산출물

### Decision
`id`, `meeting_id`, `content`, `created_by`, `source`(manual/ai_extracted),
`source_utterance_id`, `agenda_id`(어느 안건에서 나온 결정인지), `confirmed`

### ActionItem
`id`, `team_id`(액션 팀 스코프 — 회의와 분리·개인 소유 유지), `assignee_id`, `description`, `due_date`,
`completed_at`(마감 준수 산정), `difficulty`(정수 1~3 난이도 가중), `is_for_next_meeting`,
`status`(todo/in_progress/done/cancelled), `source`(manual/ai_extracted), `source_utterance_id`,
`meeting_id`·`agenda_id`(출처 추적용, nullable), `link_url`(산출물 링크), `confirmed`

> 액션 완료도(트랙 2)는 별도 스냅샷 테이블 없이 `action_items`에서 **조회 시점 라이브 계산**. ([06](06-기여도-산정.md))
>
> **변경 이유** — 액션을 회의 점수에서 분리해 별도 트랙(테스크 기여도)으로 만들며 추가. `team_id`: 테스크 기여도를 user×team으로 조인 없이 집계(meeting_id 아닌 team_id로 확정 — 회의 밖 액션도 스코프 유지, 2차 "회의 분리"와 충돌 없음). `difficulty`: 쉬운 일·어려운 일 구분 위한 난이도 가중. `completed_at`: 마감 준수(완료 시각 vs `due_date`, 마감점수 = max(0, 1−0.2×초과일수)) 계산에 필요.

## 알림

### Notification
`id`, `user_id`, `type`(meeting_soon=회의 5분 전 / action_assigned=액션 확정 시 담당자 / meeting_confirmed=회의 산출물 확정),
`title`, `body`, `meeting_id`·`action_item_id`(연관 리소스, nullable), `read`

## 측정·이벤트

### PresenceEvent
`id`, `user_id`, `meeting_id`, `event_type`(join/leave/disconnect/reconnect),
`disconnect_classification`(voluntary/involuntary — 비자발 끊김은 활성 윈도우·출석 분모서 차감),
`timestamp_offset_ms`, `reason`(user_action/network/browser_close/timeout)

> **변경 이유** — `disconnect_classification`: 네트워크 사고 등 본인 책임 아닌 끊김을 출석·발언 분모에서 빼려면 자발/사고 구분이 필요.

### AnomalyEvent
`id`, `user_id`, `meeting_id`, `event_type`(capture_loss/inference_fail/`stt_failure` 등 — stt_failure는 활성 윈도우서 차감), `timestamp_offset_ms`, `severity`, `metadata`

> **변경 이유** — `stt_failure`: 말은 했으나 음성인식 실패한 구간을 기록해 발언 비중 계산에서 제외(글자수 0으로 억울한 손해 방지). 기존 테이블 값만 추가라 스키마 변경 없음.

### ContributionScore
회의 점수(트랙 1)만 회의 단위로 저장 (UNIQUE: `user_id` + `meeting_id`).

`id`, `user_id`, `meeting_id`,
`speech_ratio`, `speech_consistency`, `attendance_ratio`, `punctuality_score`,
`meeting_score`(발언×0.6 + 참석×0.4), `confidence_level`, `excluded_indicators`

> 2트랙 구조 ([06](06-기여도-산정.md)):
> - **트랙 1 회의 점수** = 위 테이블에 저장(회의별 고정값).
> - **트랙 2 액션 완료도** = `action_items`에서 라이브 계산(스냅샷 미저장).
> - **최종 기여도** = 액션 완료도 × 0.5 + 회의 점수 누적 × 0.5 — 조회 시점 동적 계산, 별도 컬럼 없음.
> - 액션 축 필드(`action_completion_ratio`/`deadline_adherence`)는 이 테이블에서 제거되어 트랙 2로 이동.
>
> **변경 이유** — 3축 통합 → 2트랙 분리로 이 테이블이 "회의 기여도만 저장"으로 좁아짐. 액션 필드 제거(트랙 2 라이브 계산으로 이동). `composite_score`→`meeting_score`(담기는 값이 발언+참석 회의 점수뿐이라 의미 일치). `speech_consistency` 추가(발언을 양뿐 아니라 안건 참여 폭까지 측정).

## 시간 표현 규칙

- 회의 내 모든 시각은 서버 T0 기준 **상대 오프셋(`_offset_ms`)** 으로 저장
- 절대 시각은 `scheduled_at`, `t0_timestamp`, `ended_at`, `due_date`, `completed_at` 등에만 사용
