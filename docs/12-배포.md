# 12. 배포

단일 EC2(t3.small) + Docker로 운영한다. **Caddy** 한 컨테이너가 ① React 정적 빌드 서빙
② `/api` → NestJS 프록시 ③ `wss://`(socket.io) → NestJS 프록시 ④ Let's Encrypt 자동 TLS 를 모두 담당한다.
HTTPS는 필수다(`getUserMedia`·Web Speech가 보안 컨텍스트를 요구).

전반 설계는 [01-아키텍처.md](01-아키텍처.md), AWS 제약은 [10-AWS-인프라-제약.md](10-AWS-인프라-제약.md) 참고.

## 구성 요소

| 파일 | 역할 |
| --- | --- |
| `server/Dockerfile` | NestJS 멀티스테이지 빌드 → `node dist/main` (비루트, 포트 3000) |
| `client/Dockerfile` | Vite 정적 빌드 → 산출물(`dist/`)을 볼륨에 export하는 1회성 컨테이너 |
| `Caddyfile` | 정적 SPA 서빙 + `/api`·`/socket.io` 프록시 + 자동 TLS |
| `docker-compose.yml` | `client-build` → `nest` → `caddy` 3개 서비스 |
| 루트 `.env` | compose 변수(`DOMAIN`, 선택 `VITE_*`) |
| `server/.env` | 서버 런타임 변수(DB·JWT·카카오·OpenAI 등) |

**데이터 흐름:** `client-build`가 정적 산출물을 named volume `web-dist`에 복사하고 종료
→ `caddy`가 `web-dist`를 `/srv`로 서빙하고, `/api`·`/socket.io`는 `nest:3000`으로 프록시한다.
NestJS의 3000 포트는 외부에 노출하지 않는다(Caddy 내부 네트워크로만 접근).
NestJS는 Caddy를 단일 홉으로 신뢰한다(`trust proxy: 1`) — 레이트리밋 등이 프록시 IP가 아닌
실제 클라이언트 IP(X-Forwarded-For) 기준으로 동작하기 위한 전제로, Caddy 앞에 다른 프록시(CDN 등)를 두면 홉 수를 함께 조정해야 한다.

## 1. EC2 준비

- **보안 그룹 인바운드**: `22`(SSH, 본인 IP 한정 권장), `80`(HTTP, TLS 발급용), `443`(HTTPS).
  3000 포트는 열지 않는다(외부 비노출).
- Docker + Docker Compose 플러그인 설치, 저장소 클론.
- RDS MySQL은 별도(외부). EC2 보안 그룹이 RDS 보안 그룹의 인바운드에 허용돼 있어야 한다.

## 2. 도메인 · DNS

1. 도메인을 준비하고, **A 레코드**가 EC2의 공인 IP(또는 Elastic IP)를 가리키게 설정한다.
2. DNS 전파가 끝나야 Let's Encrypt 발급이 성공한다(보통 수 분~수십 분).
3. Caddy는 `DOMAIN` 설정 시 자동으로 인증서를 발급·갱신한다. 별도 certbot 불필요.

## 3. 환경 변수 작성

### 루트 `.env` (compose용)

`.env.sample`을 복사해 작성한다.

```bash
cp .env.sample .env
```

```bash
DOMAIN=mooimhacha.example.com   # A레코드를 건 실제 도메인
VITE_API_BASE_URL=              # 비우면 상대경로(동일 오리진) — 운영 권장
VITE_WS_URL=                    # 비우면 상대경로(동일 오리진) — 운영 권장
```

> 운영은 Caddy가 같은 도메인에서 `/api`·`/socket.io`를 프록시하므로 `VITE_*`를 **비워 두는 것**이 자연스럽다(상대경로 = 동일 오리진).

### `server/.env` (서버 런타임용)

`server/.env.sample`을 참고해 작성한다. 핵심 키:

```bash
PORT=3000
DB_HOST=<rds-endpoint>
DB_PORT=3306
DB_USER=<user>
DB_PASSWORD=<password>
DB_NAME=mooimhacha
JWT_SECRET=<랜덤 32+ 바이트>
KAKAO_CLIENT_ID=...
KAKAO_CLIENT_SECRET=...
KAKAO_REDIRECT_URI=https://mooimhacha.example.com/api/auth/kakao/callback
KAKAO_ADMIN_KEY=...
OPENAI_API_KEY=sk-...
CLIENT_ORIGIN=https://mooimhacha.example.com
```

> 카카오 `REDIRECT_URI`와 `CLIENT_ORIGIN`은 운영 도메인(https) 기준으로 맞추고, 카카오 개발자 콘솔의 Redirect URI 등록도 갱신해야 한다.
> `KAKAO_ADMIN_KEY`는 카카오 개발자 콘솔의 **Admin 키** — 회원 탈퇴 시 카카오 연결끊기(unlink)에 쓴다.
> 미설정이면 탈퇴는 정상 처리되지만 연결끊기만 건너뛴다(부팅 로그에 경고).

## 4. 배포 실행

```bash
cd ~/mooimhacha
git pull
docker compose up -d --build
```

- `client-build`는 빌드 후 자동 종료된다(정상). `docker compose ps`에서 `Exited (0)`이면 성공.
- `caddy`는 `client-build`가 성공적으로 끝나야(`service_completed_successfully`) 기동한다.

## 5. DB 스키마 관리 ⚠️

NestJS는 `NODE_ENV=production`에서 TypeORM `synchronize`가 **꺼져 있다**(운영 안전).
운영 스키마는 ① 최초 1회 생성(§5-1), ② 이후 마이그레이션(§5-2)으로만 변경한다.
개발 환경은 `NODE_ENV !== 'production'` 자동 동기화(synchronize)라 마이그레이션이 필요 없다.

### 5-1. 최초 1회 — 빈 RDS 스키마 생성

빈 RDS에 첫 배포할 때는 테이블이 자동 생성되지 않는다.
`server/.env`에 `DB_SYNC=true`를 넣고 한 번 기동해 스키마를 생성한 뒤, **즉시 제거**하고 재기동한다.

- `NODE_ENV=production`에서 `DB_SYNC=true`로 부팅하면 서버가 부팅 로그에 큰 경고를 남긴다(의도된 동작 — `docker compose logs nest`로 확인 가능).
- 이 경로는 **빈 DB 한정**이다. 데이터가 쌓인 운영 DB에 `DB_SYNC=true`를 켜면 synchronize가
  컬럼 rename을 drop+add로 처리하는 등 **데이터를 파괴할 수 있다.** 이미 운영 중인 DB의 스키마 변경은 반드시 §5-2 마이그레이션으로 한다.

운영 중에는 `synchronize`/`DB_SYNC`를 항상 꺼 둔다(스키마 자동 변경은 데이터 유실 위험).

### 5-2. 이후 모든 스키마 변경 — TypeORM 마이그레이션

구성 요소(설치된 TypeORM **1.0.0** CLI 기준 — DataSource 파일을 `-d`로 지정하는 방식):

| 파일/스크립트 | 역할 |
| --- | --- |
| `server/src/data-source.ts` | 마이그레이션 CLI용 DataSource. `buildTypeOrmOptions()`로 `app.module.ts`와 DB 옵션(charset·timezone·엔티티)을 공유 — 어긋나면 generate가 가짜 diff를 만든다 |
| `server/src/migrations/` | 마이그레이션 파일 보관(저장소에 커밋) |
| `npm run migration:generate / run / revert / show` | **ts 소스 기준** — ts-node(devDependency)가 필요해 로컬·빌드 스테이지 등 **개발 의존성이 설치된 환경에서만** 동작 |
| `npm run migration:run:prod / revert:prod / show:prod` | **dist 기준** — 운영 런타임 컨테이너(`--omit=dev`, dist만 존재)에서 사용 |

절차 — **운영 EC2에서** (로컬에는 DB가 없어 generate/run을 실행할 수 없다):

1. **사전 백업**(§8 runbook) — 스키마 변경 배포 전 권장.
2. **최신 코드**: `cd ~/mooimhacha && git pull` (변경된 엔티티 포함).
3. **generate** — 빌드 스테이지 이미지(개발 의존성 + ts 소스 포함)로 일회성 컨테이너 실행:

   ```bash
   cd ~/mooimhacha
   docker build --target build -t mooimhacha-server-build ./server
   docker run --rm --env-file server/.env \
     -v "$(pwd)/server/src/migrations:/app/src/migrations" \
     mooimhacha-server-build \
     npm run migration:generate -- src/migrations/<변경이름>
   ```

   현재 운영 DB와 엔티티의 **diff만** SQL로 담긴 `server/src/migrations/<timestamp>-<변경이름>.ts`가
   (볼륨 마운트를 통해) 호스트 저장소에 생성된다.

   > **baseline 마이그레이션을 손으로 작성하지 말 것.** 운영 DB는 이미 synchronize로 엔티티와
   > 일치 상태이므로 전체 스키마 baseline이 필요 없다 — 첫 generate가 '이번 릴리스의 변경분'만
   > 담은 첫 마이그레이션이 된다. (손작성 baseline을 run하면 `table already exists`로 실패한다.)

4. **검토** — 생성된 파일의 `up()`/`down()` SQL을 눈으로 확인한다. 의도한
   `ALTER TABLE ... ADD COLUMN/ADD INDEX`만 있는지, 예상 밖의 `DROP`이 섞여 있지 않은지 본다.
   문제 없으면 커밋해 저장소에 보존한다(EC2에서 push하거나 파일을 로컬로 옮겨 PR).
5. **run** — 신규 이미지를 먼저 빌드하고, **구버전이 서비스 중인 상태에서** 마이그레이션을 적용한 뒤 전환한다:

   ```bash
   docker compose build                                     # ① 신규 이미지 빌드(dist/migrations 포함)
   docker compose run --rm nest npm run migration:run:prod  # ② 구버전 서비스 중 마이그레이션 적용
   docker compose up -d                                     # ③ 신규 코드로 전환
   ```

   > **순서 주의** — 신규 코드를 먼저 띄우면(up → run) 마이그레이션 적용 전까지 신규 컬럼 조회
   > (`users.token_version` 등)로 모든 인증 요청이 실패하는 장애 창이 생긴다. 컬럼·인덱스 **추가**
   > 같은 additive 변경은 구버전 코드와 호환되므로 위 순서가 안전하다. 단 컬럼 삭제·rename 등
   > **비호환** 변경은 예외 — 짧은 중단(stop → run → up)이 필요하다.

6. **확인** — `docker compose run --rm nest npm run migration:show:prod`로 적용 여부([X] 표시) 확인,
   `docker compose logs -f nest`로 서비스 정상 확인.

되돌리기: `docker compose run --rm nest npm run migration:revert:prod` (마지막 1개씩 revert).
이미 데이터가 변형·유실된 경우에는 §8 복구 절차를 병행한다.

### 5-3. 이번 배포 시 적용 필요 — 이번 릴리스 스키마 변경분

이번 릴리스에 엔티티 변경이 포함됐다. 기존 운영 DB에는 §5-2 절차로 반영해야 한다
(운영 EC2에서 `migration:generate`를 실행하면 아래 변경분이 첫 마이그레이션 파일로 생성된다).

- `users`: `token_version`, `email_opt_out`, `is_deleted` 컬럼 추가
- `team_memberships`: `deleted_at` 컬럼 추가
- `action_items`: `meeting_id` 컬럼 추가
- 인덱스 6종 추가: `agendas(meeting_id, status)`, `utterances(agenda_id)`, `meetings(team_id)`,
  `decisions(meeting_id)`, `contribution_scores(meeting_id)`, `action_items(meeting_id)`

## 6. 로그 · 상태 확인

```bash
docker compose ps                 # 서비스 상태(client-build는 Exited 0이 정상)
docker compose logs -f nest       # 백엔드 로그
docker compose logs -f caddy      # Caddy/TLS 발급 로그
```

재배포는 다시 `git pull && docker compose up -d --build`.
클라이언트만 다시 빌드하려면 `docker compose up -d --build client-build caddy`.

## 7. 트러블슈팅

| 증상 | 원인 / 확인 |
| --- | --- |
| TLS 인증서 발급 실패 | DNS A레코드가 아직 전파 안 됨, 또는 보안 그룹에서 `80`/`443` 미개방. `docker compose logs caddy`에서 ACME 오류 확인. |
| `wss://`(socket.io) 연결 실패 | Caddy의 `/socket.io/*` 프록시 경로 확인. 클라이언트가 동일 오리진(상대경로)으로 접속하는지, `VITE_WS_URL`이 잘못 지정되지 않았는지 확인. |
| 페이지는 뜨는데 API 500/연결 거부 | `nest` 컨테이너 상태·로그 확인. `server/.env`의 DB 접속 정보, RDS 보안 그룹 인바운드 확인. |
| 새로고침 시 404 | SPA 폴백은 Caddy `try_files {path} /index.html`이 처리. `companion.html`은 실파일이라 그대로 서빙됨. |
| 카카오 로그인 실패 | `KAKAO_REDIRECT_URI`/콘솔 등록 Redirect URI가 운영 도메인(https)과 일치하는지 확인. |

## 8. 백업·복구 runbook

실데이터(회의·발화·기여도)가 쌓이는 RDS의 백업·복구 절차. 스키마 변경(§5-2) 배포 전에는
수동 백업(§8-2)을 1회 떠 두는 것을 권장한다.

### 8-1. RDS 자동 백업 확인 (1회 + 분기별 점검)

AWS 콘솔 → RDS → DB 인스턴스 선택 → **Maintenance & backups** 탭에서:

1. **Automated backups**가 `Enabled`인지 확인한다.
2. **Backup retention period**(보존 기간)를 확인한다 — `0`이면 자동 백업이 꺼진 것이다.
   **7일**로 설정을 권장한다(인스턴스 수정 → Backup retention period → 7 days).
   프리티어는 DB 스토리지 용량만큼의 백업 스토리지가 무료라 추가 비용이 없다.
3. 자동 백업이 켜져 있으면 **특정 시점 복원(point-in-time restore)** 과 일일 스냅샷이 가능하다.

확인 결과(활성 여부·보존 기간·확인 날짜)를 팀 채널이나 본 문서에 기록해 둔다.

### 8-2. 수동 백업 — mysqldump

EC2 호스트에는 mysql 클라이언트가 없으므로 docker 이미지로 실행한다.
접속 정보는 `server/.env`에서 읽는다(**비밀번호를 명령에 직접 쓰지 말 것** — 셸 히스토리에 남는다).

```bash
cd ~/mooimhacha
mkdir -p ~/backups
set -a; source server/.env; set +a    # DB_* 변수 로드

docker run --rm -e MYSQL_PWD="$DB_PASSWORD" mysql:8.0 \
  mysqldump -h "$DB_HOST" -P "$DB_PORT" -u "$DB_USER" \
  --single-transaction --no-tablespaces "$DB_NAME" \
  > ~/backups/mooimhacha-$(date +%Y%m%d-%H%M%S).sql
```

- `--single-transaction` **필수** — InnoDB 일관 스냅샷으로 덤프한다. 빼면 테이블 락이 걸려
  진행 중인 회의의 5초 RDS flush를 막을 수 있다.
- `--no-tablespaces` — RDS 사용자에게 없는 `PROCESS` 권한 요구를 피한다.
- (옵션) S3 보관: EC2 인스턴스 프로필 권한으로 `aws s3 cp ~/backups/<파일>.sql s3://<username으로 시작하는 버킷>/db-backup/`
  (버킷 규칙·권한은 [10-AWS-인프라-제약](10-AWS-인프라-제약.md) §S3·§Access Key 참고).

### 8-3. 복원

```bash
cd ~/mooimhacha
set -a; source server/.env; set +a

docker compose stop nest              # 복원 중 쓰기 차단

docker run --rm -i -e MYSQL_PWD="$DB_PASSWORD" mysql:8.0 \
  mysql -h "$DB_HOST" -P "$DB_PORT" -u "$DB_USER" "$DB_NAME" \
  < ~/backups/mooimhacha-<타임스탬프>.sql

docker compose start nest
```

RDS 스냅샷/특정 시점 복원을 쓰는 경우: 복원은 **기존 인스턴스를 덮어쓰지 않고 새 인스턴스를
만든다.** 새 인스턴스의 엔드포인트로 `server/.env`의 `DB_HOST`를 교체하고
(새 인스턴스의 보안 그룹 3306 인바운드에 EC2 보안 그룹 허용도 확인), `docker compose up -d`로 재기동한다.

### 8-4. 복구 시나리오

| 상황 | 절차 |
| --- | --- |
| 잘못된 마이그레이션 적용(데이터는 무사) | `docker compose run --rm nest npm run migration:revert:prod`로 되돌리고, 마이그레이션 파일 수정 후 재적용 |
| 마이그레이션·운영 실수로 데이터 손상 | `docker compose stop nest` → §8-3 복원(직전 수동 덤프 또는 RDS 스냅샷) → 필요 시 revert 병행 → 재기동·동작 확인 |
| RDS 인스턴스 자체 장애 | RDS 스냅샷에서 새 인스턴스 복원 → `DB_HOST` 교체 → 재기동 |

> 복원 절차는 실제 사고 전에 **1회 리허설**(테스트 덤프 → 복원)을 해 두는 것을 권장한다.
